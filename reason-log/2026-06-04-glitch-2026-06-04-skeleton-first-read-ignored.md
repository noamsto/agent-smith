# glitch-2026-06-04-skeleton-first-read-ignored

**Artifact:** /home/noams/.claude/CLAUDE.md#reading-code-skeleton-first  
**Fix type:** escalate-out-of-instructions  **Confidence:** high  **Date:** 2026-06-04

## Diagnosis

The 'Reading Code (skeleton-first)' rule already exists and is explicit (run ast-grep / code-structure / Grep before a full Read for files >~300 lines, pull full bodies only once the region is known). Yet across 87 distinct sessions and 147 incidents the agent issues unbounded full-file Read calls on files far above the threshold — 526 to 2437 lines — with no offset/limit and no preceding skeleton step. The same files (e.g. noctalia/default.nix at 640 lines) are re-read in full in separate sessions. This is a pure-prose rule that is being silently ignored at the moment of action: the model passes a bare file_path to Read and the harness happily returns the whole file. Strengthening the wording has the same failure mode as the existing imperative phrasing — the rule is read at session start but not consulted at tool-call time. The fix must make the cheap-by-default path structural, not advisory.

## Proposed change

```
Escalate to a PreToolUse hook on the Read tool that enforces skeleton-first mechanically, so the unbounded large-file read becomes impossible rather than discouraged.

Where it lives: this is a nix-store-referencing hook, so it belongs in the `--settings` overlay generated by `nix-settings-json` in `home/ai/claude-code/default.nix` (NOT settings.json — per the Settings Architecture rules, hooks that reference /nix/store paths go in the overlay). Add a `hooks.PreToolUse` entry matching `Read`, pointing at a script in the nix store (e.g. `pkgs.writeShellApplication { name = "claude-read-skeleton-guard"; ... }`).

Hook behavior (the script reads the tool_input JSON on stdin):
1. Parse `tool_input.file_path`, `tool_input.offset`, `tool_input.limit`.
2. If `offset` or `limit` is present → exit 0 (allow): a bounded read is already the intended slice-read.
3. Otherwise stat/`wc -l` the file. If line count <= 300 → exit 0 (allow): short-file exception.
4. If line count > 300 and the read is unbounded → emit a JSON decision that BLOCKS the call and returns a reason to the model, e.g.:
   {
     "hookSpecificOutput": {
       "hookEventName": "PreToolUse",
       "permissionDecision": "deny",
       "permissionDecisionReason": "<file> is <N> lines (>300). Skeleton-first: run ast-grep / the context-efficient-tools:code-structure skill / Grep for the symbol you need first, then Read with offset+limit for the region that matters. To read the whole file deliberately (security review, top-to-bottom audit, every-line-matters config), re-issue Read with an explicit limit covering the file."
     }
   }
   (deny here means 'ask the model to reconsider', surfacing the reason as a tool_result; it is not a hard user prompt.)
5. Allow an escape hatch so genuine whole-file work is not bricked: an explicit `limit` >= line count satisfies step 2, so the model can consciously opt into a full read.

Then TIGHTEN the prose rule to point at the hook so the two stay in sync — replace the 'Exceptions' line of the Reading Code section with:

  Exceptions (whole-file work — security review, top-to-bottom audit, config where every line matters): pass an explicit `limit` to Read to opt in deliberately. A PreToolUse guard (claude-read-skeleton-guard, in the --settings overlay) blocks unbounded Reads of files >300 lines and tells you to slice; bounded reads (offset/limit) and short files pass through untouched.

This converts a 'please remember' rule into a default the harness enforces, while keeping the legitimate full-read path one explicit argument away.
```

Expected effect: unbounded large-file reads stop firing; the model is nudged to slice via offset/limit or skeleton tools, while bounded reads, short files, and deliberate full reads pass through. Confidence high: consistent, high/medium-confidence evidence across many sessions all pointing at one tool, one threshold, one existing rule.
